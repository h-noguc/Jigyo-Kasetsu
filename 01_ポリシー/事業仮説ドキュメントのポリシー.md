# 事業仮説ドキュメントのポリシー

事業仮説を記述するドキュメントの構成・命名・運用ルール。

---

## フォルダ化の基準

| 段階 | ドキュメント形態 |
|------|-----------------|
| フェーズ0（アイデア） | `02_事業仮説/` 直下に単一ファイル（例: `M-Nの日程調整SaaS.md`） |
| フェーズ1（課題+WTP検証）以降 | `02_事業仮説/{事業名}/` フォルダに展開 |

アイデア段階では自由記述でよい。検証に進む段階で、下記のフォルダ構成に整理する。

---

## フォルダ構成（標準テンプレート）

フォルダ内のファイルは以下のグループ順で番号を振る。番号は連番で、グループ内のファイル数は事業ごとに異なってよい。

### グループと順序

| 順序 | グループ | 役割 | ファイル数 |
|------|---------|------|-----------|
| **① 概要** | 事業の全体地図 | コンセプト・背景仮説・提供範囲の概要・全ドキュメントへのリンク | 1（必須） |
| **② 課題と解決策** | Problem & Solution | 構造的負（誰の何が課題か）＋ 導入で何が変わるか | 1（必須） |
| **③ 事業モデル** | なぜこのモデルか | 類似モデル比較、モデル選択の論証、継続性（解約されにくい理由）、収益モデル | 1（必須） |
| **④ サービス設計** | プロダクトの中身 | 提供するサービスの具体設計。事業ごとに分割単位は異なる | 1以上（必須） |
| **⑤ 市場選定** | どの市場から攻めるか | 対象ドメイン/セグメントの候補・比較・選定 | 0〜1（任意） |
| **⑥ GTM・集客** | どう売るか | 顧客獲得チャネル、チャネル別の戦略詳細 | 1（必須） |
| **⑦ 仮説検証** | 今何をしているか | 検証方針・アクション・進捗管理。常にフォルダの末尾に置く | 1（必須） |

### 命名規則

- ファイル名は `{連番}_{内容}.md` 形式（例: `00_事業概要.md`, `03_サービス設計_業務フロー.md`）
- 連番は `00` 始まりの2桁ゼロ埋め
- サービス設計が複数ファイルになる場合は `{連番}_サービス設計_{サブ名}.md` とする
- 概要ファイル（`00_事業概要.md`）は各事業フォルダのエントリポイント

### 具体例（BPaaS事業）

```
オンライン1on1セッションBPaaS/
├── 00_事業概要.md                 ← ① 概要
├── 01_課題と提供価値.md            ← ② 課題と解決策
├── 02_事業モデル.md               ← ③ 事業モデル
├── 03_サービス設計_業務フロー.md    ← ④ サービス設計
├── 04_サービス設計_情報ストック.md   ← ④ サービス設計
├── 05_サービス設計_バックオフィス.md  ← ④ サービス設計
├── 06_ドメイン選定.md              ← ⑤ 市場選定
├── 07_GTM・集客チャネル.md         ← ⑥ GTM・集客
└── 08_仮説検証.md                 ← ⑦ 仮説検証
```

## 各ファイルの記述ルール

### 00_事業概要.md（エントリポイント）

- 事業コンセプトを1〜2段落で簡潔に記述（論証は入れない）
- 背景仮説を引用ブロックで記述
- 提供範囲の概要を表形式で記述
- フォルダ内の全ドキュメントへのリンク表（ドキュメント構成）を末尾に置く

### 仮説検証ファイル（末尾）

- frontmatter に `phase`, `owner`, `next_action`, `deadline` を記述する（`02_事業仮説/00_Index.md` の Dataview クエリが参照するため）
- 検証の方針（何を確かめるか）と、検証アクション（具体的な行動と進捗）を分離して記述する
- DM文面テンプレート等の実行資材もこのファイルに置く

### 内部リンク

- 親ファイルへのリンクは各ファイル冒頭に `> 親ファイル：[[00_事業概要]]` として記述
- 概要ファイルからの子ファイルリンクは表形式で記述（ドキュメント構成テーブル）
- Obsidian の `[[ファイル名]]` 記法を使用（フルパスは不要）

---

## フォルダ化の手順

1. 単一ファイルの内容を上記グループに分類する
2. 各グループに対応するファイルを作成し、連番を振る
3. 概要ファイルにドキュメント構成テーブルを作成する
4. 仮説検証ファイルに frontmatter を付与する
5. `02_事業仮説/00_Index.md` の事業仮説一覧を更新する
