# サービス設計：情報ストック

> 親ファイル：[[00_事業概要]]

> **ドメイン非依存**: フローと同様、蓄積すべき情報の構造はドメインに依存しない。

---

セッション事業を運営する中で蓄積される情報と、その活用。フロー（業務の流れ）が「動き」だとすると、ストックは「溜まるもの」。BPaaSとしてデータ構造の設計・維持・活用をサービスとして提供する。

単にデータを蓄積するだけでなく、**蓄積データを使って毎週の運用介入と改善を回す**ことが目的。これにより「構築して終わり」ではなく、継続的に価値が発生する。

## 受講者データ

フローの各ステップで自動的に蓄積される。蓄積した情報は、例外対応・継続フォロー・再契約提案の判断材料として運用で使い続ける。

| データ | 蓄積タイミング | 活用 |
|--------|--------------|------|
| プロフィール（氏名・連絡先・相談内容等） | ① 申込フォーム送信時 | 事前案内のパーソナライズ、初回準備、継続フォローの文脈保持 |
| 体験セッションの結果（成約/未成約/離脱） | ② 無料体験後 | 成約フォローの優先順位付け、フォロー漏れ防止、成約率改善 |
| 契約・決済情報（プラン・更新日・解約理由） | ③ 契約・決済時 / 更新時 | 請求管理、更新前フォロー、解約理由の集計と改善 |
| セッション履歴（日時・回数・次回予約状況） | ④ 各セッション後 | 未予約アラート、継続意向の早期検知、契約終了前の提案判断 |
| 対応履歴（全やりとりのログ） | 全ステップ | 例外処理時の参照、品質監査、引き継ぎ、再契約時の文脈再利用 |

## セッション記録

コーチが入力する情報。サービスとしては**入力の仕組み提供と自動整理**を担う。記録は単なるメモではなく、次回提案・継続判断・再契約導線の基礎データになる。

| データ | 内容 | 活用 |
|--------|------|------|
| 振り返りメモ | セッションで話したこと・気づき・状態変化 | 次回準備、継続意向の変化把握、個別フォロー |
| アクションアイテム | 受講者が次回までにやること・期限・達成状況 | リマインド、未達時の介入、継続提案の根拠 |
| 目標と進捗 | 初期目標、現在地、達成度、障害要因 | 成果実感の可視化、契約更新判断、卒業後フォロー設計 |
| 共有資料 | セッション中に渡した資料・リンク | 受講者がいつでも参照できる状態の維持、再契約時の再活用 |

## 事業指標

フローの各ステップの実績から自動集計される。重要なのは、可視化で終わらず「誰にいつ介入するか」を判断できる指標にすること。

| 指標 | 算出元 | コーチにとっての意味 |
|------|--------|---------------------|
| 申込数 | ① 申込フォーム | 集客の量 |
| 体験→成約の転換率 | ②→③ の遷移 | セッションの訴求力 |
| 継続率/離脱率 | ④ のループ回数 | 提供価値の維持状況 |
| 更新率（契約更新率） | 契約更新データ | 構築後も価値が発生しているか |
| 未予約期間（日数） | 最終セッション日と次回予約日 | 離脱予兆の早期検知 |
| フォロー実施率 | フォロー対象件数に対する対応件数 | 運用品質の維持 |
| 解約理由の内訳 | 解約時ヒアリング・フォーム | 改善優先度の明確化 |
| 再契約率 | 卒業後フォローからの再成約 | 継続運用価値の検証 |
| 受講者あたり売上 | 契約・決済データ | 収益性の把握 |

## BPaaSとしての提供価値

コーチが自分でやろうとすると：
- CRMの項目設計を自分で考える必要がある
- データの入力・整理を手動でやり続ける必要がある
- 指標の集計を自分でスプレッドシートで組む必要がある
- セッション記録がバラバラの場所に散在する

BPaaSが提供するもの：
- **データ構造の設計・維持** — コーチは項目設計を考えなくてよい
- **自動蓄積** — フローの各ステップで自動的にCRMに記録される
- **週次運用ダッシュボード** — 更新率・未予約期間・フォロー実施率・売上を毎週確認できる
- **セッション記録の仕組み** — コーチが記録するだけで受講者ごとに時系列整理され、次回提案に接続される
- **運用介入の起票** — 離脱予兆（未予約・未達・満足度低下）を検知し、フォロータスクを自動起票する
- **保守と継続改善** — テンプレ・フロー・指標定義を運用しながら更新し、改善を継続する
- **蓄積データの活用** — 離脱しやすいパターン、成約しやすいパターン、再契約パターンを分析する

## 継続価値が発生し続ける理由（ストック編）

情報ストックは「一度作って終わり」の資産ではない。運用し続けるほど価値が上がる構造を持つ。

| 要因 | 内容 |
|------|------|
| データ鮮度の維持が必要 | 受講者状態は毎週変化するため、更新されないデータはすぐに使えなくなる |
| 運用知見が累積する | 例外対応・フォロー履歴が増えるほど、対応品質と再現性が上がる |
| 改善ループが回る | 可視化→介入→効果測定を繰り返すことで、更新率・再契約率が改善する |
| 卒業後フォローに再利用できる | 過去の目標・進捗・やりとり履歴が、再契約時の提案精度を高める |
