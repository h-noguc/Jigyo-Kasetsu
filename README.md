## 仮説検証フレームワーク

事業仮説は以下のフェーズで管理します：

| Phase | 目的 | 主なアクション |
|-------|------|---------------|
| 0 | アイデア | 仮説の言語化、構造的負の整理 |
| 1 | 課題 + WTP | 課題の存在確認、支払意思の検証 |
| 2 | 再現性 | 複数顧客での課題検証 |
| 3 | 解決可能性 | MVP構築、パートナー確保 |
| 4 | 事業化 | スケールに向けた投資判断 |

詳細は `01_ポリシー/仮説検証のポリシー.md` を参照してください。

## 事業仮説の記述フォーマット

各事業仮説は以下の構成で記述します：

1. **用語の認識合わせ** - 主要用語の定義、ターゲットの明確化
2. **構造的負** - 解決すべき課題、なぜその課題が存在するか
   - アナロジー - 同様の課題を解決した他社事例
3. **事業概要** - 提供価値、収益モデル、差別化ポイント
4. **仮説検証の方針** - 具体的な検証アクション

## 補助ツール (.Skills)

### Markdown → PPTX 変換

LLMが生成したMarkdownを編集可能なPowerPointファイルに変換します。

```bash
cd .Skills/markdown_to_pptx
pip install python-pptx
python simple_markdown_to_pptx.py input.md output.pptx
```

### PDF → Markdown 変換

PDFファイルからテキストを抽出し、構造化されたMarkdownに変換します。

```bash
cd .Skills/pdf_to_markdown
pip install pdfplumber
python simple_pdf_to_markdown.py input.pdf output.md
```

## 運用ルール

### 仮説ごとにオーナーを明確にする
- 各仮説に1名のオーナーをアサイン
- 全員が全部見ると誰も責任を持たない

### 進捗の可視化は軽量に
週次で以下の3点だけ共有：
1. 現在のフェーズ
2. 次のマイルストン
3. ブロッカー

### 撤退判断は定例で行う
- 個人判断だと撤退が遅れがち
- 定例で「撤退検討枠」を設ける

## Obsidianについて

このボルトはObsidianで閲覧・編集することを前提としています。
- `[[リンク]]` 形式の内部リンク

- YAML frontmatterによるメタデータ管理（phase, owner, next_action, deadline）
